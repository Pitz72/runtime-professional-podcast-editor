Questa release rappresenta una **riscrittura strutturale** del progetto. ## 🔴 Fix Critici

### 1. Sistema CSS Completamente Riscritto
**Problema:** L'app caricava Tailwind CSS da CDN (`https://cdn.tailwindcss.com`). In un'app Electron impacchettata (protocollo `file://`), il CDN non è raggiungibile. L'interfaccia si renderizzava **senza alcuno stile**.

**Soluzione:**
- Installato `tailwindcss@3`, `postcss`, `autoprefixer` come dipendenze locali
- Creato `tailwind.config.js` con content pointing a `src/renderer/**/*.{ts,tsx,html}`
- Creato `postcss.config.js` per la pipeline PostCSS
- Creato `src/renderer/index.css` con direttive `@tailwind` + CSS custom (scrollbar dark theme, reset app, transizioni)
- Rimosso il tag `<script src="https://cdn.tailwindcss.com">` da `index.html`
- Rimosso il blocco `<script type="importmap">` obsoleto (era un residuo di quando l'app era browser-only)
- Aggiunto import CSS in `index.tsx`

**File modificati:** `index.html`, `index.tsx`
**File creati:** `tailwind.config.js`, `postcss.config.js`, `src/renderer/index.css`

---

### 2. AudioContext Leak Eliminato
**Problema:** Un `AudioContext` veniva creato globalmente alla riga 8 di `audioUtils.ts` (`export const audioContext = new AudioContext()`). I browser bloccano AudioContext creati senza interazione utente. Inoltre, 3 funzioni (`resampleAudioBuffer`, `normalizeAudioBuffer`, `validateAudioFile`) creavano `new AudioContext()` senza mai chiuderli, causando memory leak.

**Soluzione:**
- Rimossa la creazione globale di AudioContext (riga 8)
- Rimossa la funzione `initAudioWorklet()` (35 righe) che dipendeva dal context globale
- `resampleAudioBuffer()` e `normalizeAudioBuffer()` ora usano `OfflineAudioContext` per creare buffer, che non richiede interazione utente e non causa leak
- L'AudioContext di playback viene gestito esclusivamente da `useAudioEngine` (lazy init su interazione)

**File modificati:** `src/renderer/services/audioUtils.ts`

---

### 3. Export FLAC/AAC — Onestà nell'UI
**Problema:** L'ExportDialog e i TransportControls offrivano FLAC e AAC come opzioni di export. Selezionandoli, il sistema esportava silenziosamente un WAV (`console.warn('FLAC/AAC not implemented')`). L'utente credeva di avere un FLAC ma riceveva un WAV.

**Soluzione:**
- `TransportControls.tsx`: opzioni FLAC e AAC ora sono `disabled` con label "coming soon"
- `ExportDialog.tsx`: riscritto completamente, FLAC/AAC visivamente disabilitati e non selezionabili
- I formati funzionanti (WAV, MP3) hanno descrizioni chiare

**File modificati:** `TransportControls.tsx`, `ExportDialog.tsx`

### 4. ExportDialog — Riscrittura Completa
**Problema:** `ExportDialog.tsx` importava:
- `useAppStore` (lo Zustand store, mai usato dall'app)
- `useLogger` (hook morto)
- `useAudioExport` (hook che **non esisteva** nel codebase!)

Il componente non poteva funzionare. Era broken by design.

**Soluzione:**
- Riscritto da zero (da 362 righe a 107 righe)
- Accetta `project`, `onExport`, `isExporting` come props dal componente padre
- Nessuna dipendenza da codice morto
- UI pulita con selezione formato e info progetto

**File riscritto:** `src/renderer/components/ExportDialog.tsx`

## �️ L'Odissea della Build (Cronache di un Packaging)

Il vero banco di prova della v0.5.0 è stato il passaggio dalla modalità "dev" al pacchetto finale di Electron. Durante la creazione dell'installer (e al suo primo avvio), sono emersi problemi architetturali profondi che hanno richiesto interventi mirati:

### Errore 1: Collisione Vite + tsc nel `build` script
**Problema:** L'installer falliva in fase finale segnalando che il file `app.asar` era corrotto e `dist/main/main.js` non esisteva.
**Causa:** Lo script `build` in `package.json` eseguiva prima `build:main` (TypeScript) e poi `build:renderer` (Vite). Tuttavia, Vite per impostazione predefinita esegue l'`emptyOutDir`, svuotando brutalmente tutta la cartella `dist/` e cancellando i file del main process appena compilati.
**Soluzione:** Inversione dell'ordine di compilazione in `package.json`: prima il renderer, *poi* il main process, salvando l'output di `tsc`.

### Errore 2: Conflitto ES Module vs CommonJS ("exports is not defined")
**Problema:** L'app installata crashava istantaneamente all'avvio con `ReferenceError: exports is not defined in ES module scope`.
**Causa:** Il progetto usa `Vite`, che richiede `"type": "module"` nel `package.json` root. Questo obbliga Node.js (e quindi Electron) a trattare **tutti** i file JavaScript come moduli ES. Tuttavia, sia lo script `preload.js` che parti fondamentali per il `webPreferences` di Electron si aspettano un tradizionale ambiente **CommonJS**. Quando l'app partiva, Electron iniettava variabili (`exports`) tipiche del vecchio standard, mandando il motore V8 in tilt davanti a un "falso" modulo ES.
**Soluzione architetturale elegante:**
Rinominati i file sorgente `main.ts` e `preload.ts` in **`main.cts`** e **`preload.cts`**. L'estensione `.cts` forza esplicitamente il compilatore TypeScript a emettere file con estensione **`.cjs`** (CommonJS puro). Electron li riconosce e li esegue perfettamente ignorando il `"type": "module"` globale, senza richiedere acrobatiche configurazioni del bundler. Contestualmente è stata sostituita la libreria `electron-is-dev` (incompatibile con questo workflow) con l'API nativa `!app.isPackaged`.

### Errore 3: Blocco CSP sui file Audio ("Refused to connect")
**Problema:** Al primo avvio dell'app funzionante, il drag & drop di un file audio nel "file bin" falliva silenziosamente (o stampava un alert rosso) con errore *CSP (Content Security Policy)*: `Violates the directive: "connect-src 'self'..."`.
**Causa:** Per difendere l'app dalle solite vulnerabilità Electron, era stata impostata una severa `Content-Security-Policy` nel tag `<meta>` di `index.html`. L'audio engine, per bypassare le limitazioni di path di sistema imposte dal browser locale, per i file leggeri utilizza l'API `FileReader` convertendoli in URL `data:audio/wav;base64,...` passati poi alla funzione Web Audio API `fetch()`. La mia CSP stringente proibiva a `fetch()` di comunicare con URL tipo `data:` e `blob:`.
**Soluzione:** Aggiornata la CSP nel blocco `<head>` estendendo il `connect-src`:
`connect-src 'self' https://generativelanguage.googleapis.com data: blob:;`
